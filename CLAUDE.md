# CLAUDE.md тАФ Project Guide for AI Assistants

## Project Overview

Bhaskar's Toolkit тАФ a personal productivity suite hosted on Cloudflare Pages. Currently one live app (Vibe Planner), with more planned. Zero dependencies, vanilla JavaScript, single HTML file per app.

## Tech Stack

- **Frontend:** Vanilla JS (ES5-compatible), CSS custom properties, no build tools
- **Backend:** Supabase (PostgreSQL + Auth + Row Level Security)
- **Hosting:** Cloudflare Pages (auto-deploys from GitHub `main` branch)
- **CDN:** Supabase JS SDK loaded from `cdn.jsdelivr.net`

## Key Architecture Decisions

- **IIFE pattern** wraps all JS to avoid global scope conflicts (especially `window.supabase` from the CDN)
- **Dual mode:** Cloud (Supabase) when `config.js` has real credentials, Local (localStorage) otherwise. Controlled by `IS_CLOUD` flag.
- **Supabase client** is stored as `var db` (never `supabase`) to avoid naming conflict with the SDK global
- **UUIDs** are generated by Supabase on insert (not client-side) to match the `UUID` column type
- **RLS policies** enforce per-user data isolation тАФ each user only sees their own vibes

## Commands

There is no build step for local development. Open `index.html` in a browser.

### Testing


## Database Schema

Table: `vibes` with columns: `id` (UUID, auto-generated), `user_id` (UUID, references auth.users), `text`, `status`, `category`, `time_spent` (integer, minutes), `notes` (text), `scheduled_date`, `created_at`. Full SQL in `docs/SUPABASE_SETUP.md`.

## Statuses

Four statuses: `spark` (new idea), `in_progress`, `scheduled` (has a date), `done`. The old "deferred/future" status was removed тАФ use Spark for "someday" items.

## Important Gotchas

1. **Never commit `config.js`** тАФ it's gitignored. Use `config.example.js` as template.
2. **Supabase SDK global:** The CDN creates `window.supabase`. Never declare `let supabase` or `const supabase` тАФ use `db` instead.
3. **UUID IDs:** The `vibes` table uses UUID primary keys. Let Supabase generate them on insert (don't send `id` field).
4. **Cloudflare Pages, not Workers:** This is a static site. Don't use Wrangler тАФ it respects `.gitignore` and will skip `config.js`.
5. **ES5 style:** The codebase uses `var`, `.then()`, and `function()` тАФ not `const`/`let`, `async`/`await`, or arrow functions.

## Security

### Signup Control
- **Signups are disabled by default** via `SIGNUP_ENABLED: false` in `config.js` / Cloudflare env vars
- To create a new account: temporarily set `SIGNUP_ENABLED: true`, create the account, then set it back to `false`
- Also disable signups server-side in Supabase Dashboard: **Authentication тЖТ Settings тЖТ toggle OFF "Allow new users to sign up"**

### Rate Limiting
- Client-side rate limiter: max 5 auth attempts per 60-second window (login + signup)
- Supabase also enforces server-side rate limits on auth endpoints

### Auth Error Sanitization
- Raw Supabase error messages are never shown to users тАФ prevents email enumeration attacks
- Errors are mapped to generic messages (e.g., "Invalid email or password")

### Content Security Policy
- CSP meta tag restricts scripts to `self` + jsdelivr CDN, connections to `self` + `*.supabase.co`
- Prevents inline script injection from external sources

### Other Protections
- **XSS-safe DOM**: Uses `h()` builder with `createTextNode()` тАФ never `innerHTML` for user content
- **RLS enforced**: All Supabase CRUD includes `user_id` filter belt-and-suspenders with RLS policies
- **Supabase SDK pinned** to specific version (`@2.49.4`) to prevent supply chain attacks via CDN
- **config.js gitignored**: Real credentials never enter version control

### Supabase Dashboard Hardening (Manual Steps)
1. **Authentication тЖТ Settings**: Disable "Allow new users to sign up" for production
2. **Authentication тЖТ Rate Limits**: Verify rate limits are enabled (default: on)
3. **Authentication тЖТ Providers тЖТ Email**: Enable "Confirm email" for production use
